Java 开发团队手册

第一章 编码规范

第一条 命名规范
1. 类名使用大驼峰（PascalCase）：UserService、OrderController
2. 方法名和变量名使用小驼峰（camelCase）：getUserName、maxRetryCount
3. 常量使用全大写加下划线：MAX_CONNECTIONS、DEFAULT_TIMEOUT
4. 包名使用全小写：com.smartdoc.api、com.smartdoc.service
5. 布尔变量以 is/has/should 开头：isActive、hasPermission、shouldRetry

第二条 代码格式
1. 缩进使用 4 个空格，不使用 Tab
2. 单行代码不超过 120 个字符
3. 方法之间空一行
4. 花括号不另起一行（K&R 风格）

第三条 注释规范
1. 每个 public 类和方法必须有 JavaDoc
2. JavaDoc 应描述"做什么"而非"怎么做"
3. 复杂逻辑必须添加行内注释
4. TODO 注释格式：// TODO(姓名): 描述

第二章 API 设计指南

第四条 RESTful 设计原则
1. URL 使用名词复数：/api/users、/api/orders
2. 使用 HTTP 方法表达操作：GET 查询、POST 创建、PUT 更新、DELETE 删除
3. 使用合适的 HTTP 状态码：200 成功、201 创建、400 请求错误、401 未认证、403 禁止、404 未找到、500 服务器错误
4. 分页参数：page 和 size，默认 page=0, size=20
5. 排序参数：sort，格式为 sort=field,direction（如 sort=name,asc）

第五条 请求与响应格式
1. 请求和响应统一使用 JSON 格式
2. 响应体统一包装：{ "code": 200, "message": "success", "data": {...} }
3. 错误响应包含错误码和详细信息：{ "code": 400, "message": "参数校验失败", "details": [...] }
4. 日期时间使用 ISO 8601 格式：2026-05-11T10:30:00Z
5. 金额使用字符串表示，避免浮点精度问题

第六条 接口版本管理
1. API 版本放在 URL 路径中：/api/v1/users
2. 重大变更升级主版本号（v1 → v2）
3. 旧版本至少保留 6 个月
4. 版本迁移需提供迁移文档

第三章 部署流程

第七条 环境管理
1. 开发环境（dev）：开发人员本地调试
2. 测试环境（test）：QA 团队功能测试
3. 预发环境（staging）：模拟生产环境验证
4. 生产环境（prod）：对外提供服务
5. 所有环境配置通过环境变量注入，不硬编码

第八条 CI/CD 流程
1. 代码提交到 feature 分支触发自动化测试
2. 合并到 main 分支触发自动部署到测试环境
3. 打 tag 触发部署到预发环境
4. 手动审批后部署到生产环境
5. 部署后自动执行健康检查

第九条 数据库变更管理
1. 所有数据库变更必须通过迁移脚本（Flyway）
2. 迁移脚本命名规则：V{版本号}__{描述}.sql
3. 禁止直接修改生产数据库
4. 迁移脚本必须向前兼容（支持回滚）
5. 大表变更需要在维护窗口执行

第四章 代码审查

第十条 审查清单
1. 代码是否符合编码规范（第一条至第三条）
2. 是否有明显的安全漏洞（SQL 注入、XSS 等）
3. 错误处理是否完善
4. 是否有足够的单元测试（覆盖率 ≥ 70%）
5. 接口设计是否符合 API 设计指南（第四条至第六条）

第十一条 审查流程
1. PR 至少需要 1 位 Reviewer 批准
2. 核心模块需要 Tech Lead 批准
3. CI 全部通过后才能合并
4. 合并后自动删除分支

第十二条 安全审查要点
1. 敏感信息不能硬编码（密码、密钥等使用环境变量）
2. 用户输入必须校验和转义
3. API 接口必须有认证和授权
4. 日志中不能记录敏感信息（密码、Token）
5. 第三方依赖需定期检查已知漏洞
